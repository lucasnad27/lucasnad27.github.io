1 cluster management

1.1 interactive cluster (always on)

1.1.1 advantages

  1. zero latency
  2. simple!
  3. the primary disadvantage can be minimized with autoscaling
    1. optimized autoscaling
      • Scales up from min to max in 2 steps.
      • Can scale down even if the cluster is not idle by looking at shuffle file state.
      • Scales down based on a percentage of current nodes.
      • On automated clusters, scales down if the cluster is underutilized over the last 40 seconds.
      • On interactive clusters, scales down if the cluster is underutilized over the last 150 seconds.
    2. standard autoscaling
      • Starts with adding 4 nodes. Thereafter, scales up exponentially, but can take many steps to reach the max.
      • Scales down only when the cluster is completely idle and it has been underutilized for the last 10 minutes.
      • Scales down exponentially, starting with 1 node.

1.1.2 disadvantages

  1. potential for unused resources when we aren't running counts
  2. cost of always having a driver running, even with autoscaling

1.2 on demand (cluster configuration linked to a job type)

1.2.1 advantages

  1. could create custom setups for different types of customers
    1. e.g., LA runs a giant cluster while Mill Valley runs on a tiny cluster
  2. only pay for AWS & DBUs while we are running jobs
  3. simple!

1.2.2 disadvantages

  1. latency of 2-5 minutes

1.3 pool cluster (cluster pulls from a set of always-on AWS resources)

1.3.1 advanatages

  1. great if we start to diversify our use of databricks (e.g., for more machine learning models, business analysis, etc.,
  2. no DBUs charged (databricks pricing)
  3. no latency!

1.3.2 disadvantages

  1. we are still charged for AWS usage, which is the bulk of our cost for these kinds of jobs
  2. since this is our first usage of databricks, we aren't able to fully utilize pooling functionality
